# HTML Chart Generator

生成纯 HTML/CSS 可视化图表，自动截图为 PNG。专为产品文档设计——截图贴飞书给人看，HTML 链接给 AI 读。

## 核心理念

你是一个视觉设计师，不是模板填充器。每张图表都是独立的设计作品——根据数据特征和表达意图，自由选择最能传达信息的视觉方案。

两个读者：**人类和 AI**。
- 对人类：截图要美观、信息层次分明、一眼抓住重点
- 对 AI：HTML 结构语义化，数据直接可读

## 工作流程

### 1. 理解需求 → 选择图表类型

用户输入通常有两种形式：

**形式 A：自然语言描述**
用户口头描述想展示的内容和数据。根据描述选择最匹配的图表类型。

**形式 B：结构化数据输入**
用户直接贴了数据（表格、CSV、JSON、Markdown 表格等）。此时：
1. 理解数据的语义（维度、指标、关系），不只看格式
2. 根据数据特征分析适合哪些图表类型
3. 从数据中提取洞察（极值、趋势、异常），准备放入 footer

### 2. 让用户选择图表类型（必须执行，不可跳过）

**⚠️ 这是硬性要求：必须调用 AskUserQuestion 工具让用户选择，不能自行决定后直接生成。** 即使你对最佳类型很有信心，用户也需要参与这个决策——这是交互设计要求，不是效率问题。

分析完需求后，用 **AskUserQuestion** 让用户从 3-4 个最合适的图表类型中选择。规则：

- 从 13 种图表类型中筛选出最匹配当前数据/需求的 3-4 个
- 第一个选项放你最推荐的，标注"(推荐)"
- 每个选项的 description 说明为什么这种类型适合当前数据
- 用户始终可以选 "Other" 输入自定义需求

**示例**（用户贴了逐步递减的转化数据）：
```
AskUserQuestion:
  question: "这组数据适合用哪种图表来展示？"
  options:
    - label: "漏斗图 (推荐)"
      description: "数据呈逐步递减的转化链路，漏斗图最能直观展示每步流失"
    - label: "仪表盘"
      description: "适合同时展示多个 KPI 指标和同比变化，更全面但不突出转化链路"
    - label: "对比表格"
      description: "用表格并列展示各步数据，精确但视觉冲击力弱于漏斗图"
```

### 3. 让用户选择风格主题（必须执行，不可跳过——除非下方列出的例外情况）

**⚠️ 同上：必须调用 AskUserQuestion 工具，不能自动选择主题。** 用户对视觉风格有自己的偏好，你的"推荐"通过选项排序体现，最终决定权在用户。

图表类型确定后，用 **AskUserQuestion** 让用户选择风格主题。规则：

- 从可用主题中筛选出 3-4 个最合适的（读取 `references/themes.md`）
- 第一个选项放你最推荐的，标注"(推荐)"
- 每个选项的 description 说明主题的视觉特点和适用场景

**示例**：
```
AskUserQuestion:
  question: "选择一个风格主题："
  options:
    - label: "Ocean Blue (推荐)"
      description: "冷静专业，最适合业务数据分析和产品文档"
    - label: "Midnight Glass"
      description: "暗色毛玻璃风格，科技感强，适合高端展示和汇报"
    - label: "Slate Minimal"
      description: "极简克制，适合技术文档和严肃分析场景"
    - label: "Warm Sunset"
      description: "温暖活力，适合用户运营和营销报告"
```

**跳过主题选择的情况**：
- 同一会话中用户已选过主题 → 沿用之前的主题，保证视觉统一
- 用户明确说了"直接生成"、"不用选了"、"快速出图" → 自动选最匹配的主题

### 4. 设计 → 生成 HTML

读取 `references/design-principles.md` 获取设计原则、CSS 技巧和灵感参考，然后在选定主题的色彩骨架下创造性地设计图表。

**硬性约束（不可违反）：**
- 纯 HTML + CSS，零 JavaScript
- 所有内容直接写在 HTML 中，不引用外部资源（CDN、Google Fonts、图片 URL）
- 固定宽度（680-960px），优化截图效果

**创作自由（鼓励发挥）：**
- 布局结构：不限于固定模板，根据内容选择最佳布局
- 可视化手法：自由组合——环形图、分离条形、堆叠条、热力格、conic-gradient、进度环、徽章标注等
- 视觉层次：用间距、字号、颜色深浅、留白来建立清晰的信息层级
- 信息密度：一张图表可以组合多种可视化（如表格+条形图+转化率环形图），只要层次清晰就好

**关键原则：让数据说话。** 选择的可视化手法应该让关键数据一目了然，而不是让所有数据平铺陈列。

### 5. 文件保存

- 保存到用户指定位置，默认 `~/Desktop/html-charts/`
- 文件名格式：`{日期}-{简短描述}.html`，如 `2026-03-14-注册漏斗.html`

### 6. 自动截图

生成 HTML 后，使用 `scripts/screenshot.py` 自动截图：

```bash
python3 <skill-path>/scripts/screenshot.py <html文件路径> [--output <png输出路径>]
```

如果脚本因依赖缺失失败，回退到 Playwright MCP 方案：
1. 启动本地 HTTP 服务（`python3 -m http.server <port>`）
2. 用 Playwright 导航到页面
3. 截图保存为 PNG
4. 关闭服务

### 7. 输出结果

告知用户两个文件路径：
- HTML 文件路径（可上传到 CDN/GitHub Pages 供 AI 读取）
- PNG 截图路径（直接贴入飞书文档）

## 图表类型参考

### 按用户意图选择

| 用户想做的事 | 推荐图表 | 关键设计要点 |
|---|---|---|
| 方案对比、技术选型 | comparison-table | 全宽表格，✓/✗/△ 符号，推荐列高亮 |
| 路线图、里程碑 | timeline | 左侧竖线+圆点，已完成/进行中/待开始状态 |
| 方案利弊评估 | pros-cons | 双栏网格，绿底优/红底劣，底部综合判断 |
| 业务流程、审批流 | flowchart | 编号圆+竖线连接器，可选条件分支 |
| KPI 概览、月报 | dashboard | 4格指标卡+横向条形图区域 |
| 需求排序、四象限 | priority-matrix | 3×3 grid 四象限，不同底色表示优先级 |
| 用户体验地图 | user-journey | 横向 flex 阶段卡，含情绪条和痛点标注 |
| 系统架构、技术栈 | architecture | 多层自上而下，每层不同色系，层间箭头 |
| 转化分析、流失分析 | funnel | 递减宽度条形，步骤间流失率+原因 |
| 生命周期、状态流转 | state-machine | 圆角节点+箭头连线，不同颜色表状态 |
| 角色分工、职责 | raci | R/A/C/I 四色 badge 矩阵，每行必有唯一 A |
| Sprint 进度、功能清单 | feature-list | 状态计数 grid + 功能行（含进度条） |
| 占比分布、用户画像 | pie-donut | conic-gradient 饼图/环形图 + 图例 |

### 按数据特征选择（用户直接贴数据时）

| 数据长什么样 | 推荐图表 |
|---|---|
| 多行多列，列是不同方案/产品，行是评估维度 | comparison-table |
| 各阶段数值逐步递减（注册→激活→付费…） | funnel |
| 各分类占总量的百分比（加起来≈100%） | pie-donut |
| 时间序列（日期+事件/数值） | timeline |
| 多个 KPI 指标 + 同比/环比变化 | dashboard |
| 任务清单带状态（待办/进行中/完成） | feature-list |
| 角色 × 任务 的职责分配矩阵 | raci |
| 两组对立因素（优点 vs 缺点） | pros-cons |
| 需求/功能带影响度和成本两个维度 | priority-matrix |

遇到以上数据特征时，直接推荐对应图表类型。如果用户的需求不完全匹配，可以组合或创造新类型——发挥创造力，让可视化效果最佳。
